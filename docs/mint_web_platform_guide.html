What is Mint Web?

A design-to-code platform — draw your UI visually, then Mint generates production-ready full-stack code with a real database and API baked in.

Visual Canvas

Design your UI with a powerful editor. Every element maps directly to real code.

Mint Runtime

Lightweight engine managing state, actions, and data bindings between UI and backend.

Auto Backend

PostgreSQL database and CRUD API endpoints generated instantly for your tables.

Workflows

Build complex business logic using a node-based visual flowchart editor — no code needed.

Live Sync

Push canvas changes to your running app instantly — no rebuild, no redeploy.

Multi-Framework

Export to React, Next.js, Vue, Svelte and more. You own the code.

Mental model: Think of Mint as three layers — the Canvas (what the user sees), the Runtime (the logic layer), and the Backend (data). All three stay in sync automatically.

Everything starts on the canvas. Each Frame becomes a route in your app. Properties set in the sidebar map 1-to-1 with CSS/layout rules.

Mint Runtime — state, bindings & actions

The Runtime is the brain of your app. It holds data in State, connects that data to UI elements via Bindings, and runs logic via Actions triggered by events.

State management

Global

Accessible from any screen. Use for data that persists across routes — e.g. currentUser, auth tokens. Enable Persist to write to localStorage automatically.

Local

Scoped to a single frame/component. Use for UI-only state like isLoading, modalOpen, form field values. Destroyed when the frame unmounts.

Data bindings

Select a component

Click any element (e.g. a Text box or a Button label).

Open Bindings panel

Click the link icon next to the property you want to connect (e.g. "Content", "Color", "Visible").

Enter an expression

Use $global.userName for global state or $local.count for local state. Full expressions work: $local.itemCount > 0 ? 'Items' : 'Empty'

Actions — the four types

setState

Update state data

Writes a value to a specific path in your state tree. $args.0 means "the first argument passed when this action was called".

{ "path": "todos", "value": "$args.0", "also": "SET $lastUpdated = now()" }
also lets you update a second state variable in the same action — useful for timestamps or counters.
fetch

HTTP requests & DB queries

Calls an API endpoint. storePath automatically saves the response into that state key, so you don't need a separate setState afterwards.

{ "url": "/api/todos", "method": "POST", "headers": { "Content-Type": "application/json" }, "body": { "title": "$args.0", "status": "pending" }, "storePath": "todos", "onSuccess": "CALL loadTodos; TOAST 'Saved!'" }
onSuccess chains multiple actions with semicolons. TOAST shows a notification; CALL runs another named action.
navigate

Change screens

Pushes a new route. Pass dynamic params to send data to the destination screen — access them there via $route.params.userId.

{ "route": "/dashboard", "params": { "userId": "$global.user.id" } }
condition

Branching logic

Evaluates an expression and runs a different action list depending on the result. Each entry in then / else is a command string.

{ "expression": "$global.isLoggedIn", "then": ["CALL goToDashboard"], "else": ["OPEN_MODAL 'LoginModal'"] }

Workflows (Flows)

When action JSON strings get complex, switch to Flows — a visual node-based editor that lets you design multi-step logic as a flowchart.

Open the Workflows tab

Click the Workflows tab in the left sidebar of the editor.

Add a workflow

Click Add Workflow, give it a name. This workflow can then be called from any action using CALL myWorkflowName.

Add nodes

Three categories of nodes are available:

Logic: Condition, Loop, Delay Backend: API Call, DB Query UI: Navigate, Show Modal

Connect with edges

Draw arrows between nodes to set the execution order. Click any arrow to add a conditional expression on it, e.g. {{$_node_1_output.success}} — the path only executes if the condition is truthy.

When to use Flows vs Actions: Use action JSON for simple, single-step logic. Switch to Flows when you need branching, loops, or chaining more than 3 steps — the visual graph is much easier to debug.

Database & backend

Mint provisions a real PostgreSQL database for your project automatically. You define tables in a UI, Mint generates the CRUD API routes. No backend code needed.

Enable the database

Go to the Database tab → click Enable Database. Mint provisions a PostgreSQL instance scoped to your project.

Create a table

Click New Table. Define each column with a name, type, and optional constraints.

UUIDid — primary key, auto-generated
Texttitle, description, email…
Booleancompleted, isActive…
TimestampcreatedAt, updatedAt…

Set relations

Click the Relations tab on any table. Define one-to-many (e.g. a user has many todos) or many-to-many (e.g. posts ↔ tags). Mint generates the join table and foreign keys for you.

Set Row Level Security (RLS)

Under Policies, add rules to control who can read or write each row. Example policy for "user can only see their own rows":

auth.uid() = user_id

Mint applies this as a PostgreSQL RLS policy automatically — no SQL migration files needed.

Accessing data: Once your table exists, call it from a fetch action using /api/todos (Mint maps table names to route paths). Use the storePath field to auto-save the result into state. For raw SQL queries, use a DB Query node in a Workflow — just write the table name as defined; namespacing is handled for you.

Export, convert & live sync

When your design is ready, Mint compiles it into real source code. You can download it, push it to GitHub, or pull it with the CLI — and keep the canvas and running app in sync.

How conversion works

Click Export

The Export button is in the top header. Mint "flattens" your canvas design tree into optimized component code.

What gets generated

Mint outputs: a MintProvider wrapper component (handles the runtime), all your UI components with correct props, backend API routes for each table, and a config file.

How to extract your code

Download ZIP

Get the full source code as a ZIP. Unzip and run — it's a complete project in your chosen framework.

Push to GitHub

Connect your GitHub repo in settings. Mint pushes changes directly on each export.

CLI — npx mint-cli pull

Run in an existing project to pull the latest canvas state as code. Useful for teams where designers work in Mint and developers own the repo.

npx mint-cli pull

Live Sync — update without rebuilding

Make a canvas change

Change anything — a button color, a font size, a layout. These are "design-layer" changes Mint can push without recompiling logic.

Click "Commit Changes"

In the editor header. Mint publishes a new version manifest.

Sync Daemon picks it up

Your running app (with the Sync Daemon enabled) polls /api/sync and detects the new version. It hot-patches the UI — no page reload.

How Sync Daemon works under the hood: The Daemon is a lightweight service worker embedded by MintProvider. It polls /api/sync every few seconds. When the version hash changes, it fetches the updated component definitions and re-renders only the affected parts of the tree — similar to React Fast Refresh but driven from the cloud canvas.

Pro tips for building in Mint

Things that will save you time once you know them.

Use $ everywhere in expressions

Any binding or action field accepts a $ expression. $global.user.name, $local.count + 1, $route.params.id — all work. Ternary expressions like $local.error ? 'red' : 'inherit' are valid for color bindings.

Don't prefix table names in raw SQL

The Runtime handles namespacing. If your table is named todos, write SELECT * FROM todos — not a prefixed internal name. Mint resolves it for you.

Use the browser console for debugging

The Mint Runtime logs every action execution and state change to the browser console. Open DevTools → Console and watch state mutations in real time as you interact with your app.

Persist global state for auth

For anything auth-related — user object, token, role — enable the Persist flag on your global state. It writes to localStorage automatically, so your user stays logged in on refresh.

Chain actions with semicolons in onSuccess

The onSuccess field in a fetch action runs a sequence. Use semicolons to chain: CALL loadTodos; TOAST 'Done!'; CALL resetForm — they run in order, left to right.

Use Flows for anything with branching

If your logic has an if/else or needs to loop, build it as a Flow instead of nesting condition actions. Flows are visual, debuggable, and much easier for teammates to understand.

Name frames after your routes

The frame name becomes the URL. UserProfile → /user-profile. Be intentional with naming from the start — renaming frames later can break navigation actions that hardcode the route string.
